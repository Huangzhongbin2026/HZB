# 代码版本与迭代管理规范

## 1. Git 分支模型
- main：生产稳定分支，仅接收已验证发布内容。
- develop：日常集成分支，所有功能分支合并到此。
- feature/*：单个功能开发分支，例如 feature/order-urgent-crud。
- hotfix/*：线上紧急修复分支。
- release/*：发布候选分支，用于回归与发布准备。

## 2. 提交规范（Conventional Commits）
- feat: 新功能
- fix: 缺陷修复
- refactor: 重构
- docs: 文档
- test: 测试
- chore: 构建/工具链

示例：
- feat(task): 新增已下单加急任务创建接口
- fix(report): 修复看板超时统计口径

## 3. 关键节点备份策略
- 每完成一个里程碑，创建一个 tag 备份。
- 命名规则：backup/v{主版本}.{次版本}.{修订}-{日期}
- 示例：backup/v1.1.0-20260320

## 4. 发布节奏建议
- 小步快跑：每周一次迭代发布。
- 发布前必须完成：
  - 全链路回归测试
  - 数据库脚本审查
  - Changelog 更新

## 5. 合并检查清单
- 代码通过 lint 和单元测试。
- API 变更同步更新 OpenAPI 文档。
- 数据库变更同步更新 SQL 脚本和回滚方案。
- 影响范围与风险项在需求文档登记。

## 6. 分支与备份命令
```bash
# 新功能分支
git checkout -b feature/<module>-<short-name>

# 合并到 develop
git checkout develop
git merge --no-ff feature/<module>-<short-name>

# 创建关键节点备份
git tag -a backup/v1.1.0-20260320 -m "M1 架构与骨架完成"
git push origin --tags
```

## 7. 安全要求
- 禁止在仓库提交明文密码、Token、密钥。
- 使用 .env 或配置中心管理敏感信息。
- 账号权限遵循最小权限原则。
